## 问题一（致命）：WASAPI 共享模式不允许改格式

`audio_windows.cpp` 里这段是核心 bug：

```cpp
client->GetMixFormat(&pwfx);
// ❌ 共享模式下不能改这些字段
pwfx->wFormatTag = WAVE_FORMAT_IEEE_FLOAT;
pwfx->nChannels  = 1;   // ← 这行最危险
pwfx->wBitsPerSample = 32;
pwfx->nBlockAlign    = 4;
pwfx->nAvgBytesPerSec = pwfx->nSamplesPerSec * 4;
```

WASAPI 共享模式要求**原样使用** `GetMixFormat` 返回的格式，绝大多数 Windows 设备返回的是立体声（2 声道）`WAVE_FORMAT_EXTENSIBLE`。强改 `nChannels=1` 导致 `Initialize` 返回 `AUDCLNT_E_UNSUPPORTED_FORMAT`，进入 `if (FAILED(hr)) return;` 静默退出——你完全不知道线程已经死了。

**修法**：接受原始格式，读数据时手动降混到单声道：

```cpp
void capture_loop(IMMDevice* device) {
    HRESULT hr = CoInitializeEx(nullptr, COINIT_MULTITHREADED);
    bool comOwned = SUCCEEDED(hr) || hr == S_FALSE;
    if (FAILED(hr) && hr != RPC_E_CHANGED_MODE) return;

    IAudioClient* client = nullptr;
    hr = device->Activate(__uuidof(IAudioClient), CLSCTX_ALL,
                          nullptr, (void**)&client);
    if (FAILED(hr)) { if (comOwned) CoUninitialize(); return; }

    WAVEFORMATEX* pwfx = nullptr;
    client->GetMixFormat(&pwfx);
    // ✅ 不改格式，记录实际声道数
    int nChannels = pwfx->nChannels;
    bool isFloat  = (pwfx->wFormatTag == WAVE_FORMAT_IEEE_FLOAT);
    // WAVE_FORMAT_EXTENSIBLE 子类型判断
    if (pwfx->wFormatTag == WAVE_FORMAT_EXTENSIBLE) {
        auto* ext = reinterpret_cast<WAVEFORMATEXTENSIBLE*>(pwfx);
        isFloat = (ext->SubFormat == KSDATAFORMAT_SUBTYPE_IEEE_FLOAT);
    }

    REFERENCE_TIME hnsRequested = 100'000;
    hr = client->Initialize(AUDCLNT_SHAREMODE_SHARED,
                            0, hnsRequested, 0, pwfx, nullptr);
    CoTaskMemFree(pwfx);   // ← 之前漏掉了，这里顺手补上
    if (FAILED(hr)) {
        client->Release(); device->Release();
        if (comOwned) CoUninitialize();
        return;
    }

    IAudioCaptureClient* capture = nullptr;
    client->GetService(__uuidof(IAudioCaptureClient), (void**)&capture);
    client->Start();

    while (g_running.load(std::memory_order_relaxed)) {
        Sleep(5);
        UINT32 pktLen = 0;
        capture->GetNextPacketSize(&pktLen);
        while (pktLen > 0) {
            BYTE* rawData; UINT32 nFrames; DWORD flags;
            capture->GetBuffer(&rawData, &nFrames, &flags, nullptr, nullptr);

            if (!(flags & AUDCLNT_BUFFERFLAGS_SILENT)) {
                auto* src = reinterpret_cast<float*>(rawData);
                // ✅ 降混：多声道 → 单声道
                std::vector<float> mono(nFrames);
                if (isFloat) {
                    for (UINT32 i = 0; i < nFrames; ++i) {
                        float sum = 0;
                        for (int ch = 0; ch < nChannels; ++ch)
                            sum += src[i * nChannels + ch];
                        mono[i] = sum / nChannels;
                    }
                }
                // 非 float 格式（int16/int32）此处需额外转换，
                // 但现代 Windows 基本都返回 float，暂时先 assert
                g_ring->write(mono.data(), nFrames);
            }
            capture->ReleaseBuffer(nFrames);
            capture->GetNextPacketSize(&pktLen);
        }
    }

    client->Stop();
    capture->Release(); client->Release(); device->Release();
    if (comOwned) CoUninitialize();
}
```

## 问题二（重要）：线程失败无法反馈给 Dart

`capture_loop` 失败时静默退出，`platform_audio_start` 返回 0（成功），Dart 侧完全不知道。需要一个状态变量：

```cpp
// audio_windows.cpp
std::atomic<bool> g_capture_alive{false};

// capture_loop 末尾正常启动后设 true，退出时设 false
// platform_audio_start 等待短暂确认
int platform_audio_start(RingBuffer* ring) {
    // ... 启动线程 ...
    // 等待最多 500ms 确认线程存活
    for (int i = 0; i < 50; ++i) {
        Sleep(10);
        if (g_capture_alive.load()) return 0;
    }
    return -3;  // 线程启动了但 WASAPI 初始化失败
}
```

Dart 侧 `audio_engine.dart` 里的 `start()` 已经对非零返回值抛异常，补上这个返回码就能把错误传到 UI。

## 问题三（编译隐患）：`namespace` 嵌套在 `extern "C"` 里

`audio_bridge.cpp` 第 57 行附近：

```cpp
extern "C" {
    // ...
    namespace {          // ❌ extern "C" 块内不能有 namespace
        float g_freq_history[5] = {0};
        ...
    }
    EXPORT YinResult audio_poll_pitch(...) { ... }
}
```

这是 ill-formed C++，MSVC 可能接受但行为未定义。把平滑状态移到 `extern "C"` 块外面：

```cpp
// audio_bridge.cpp — extern "C" 块之外
namespace {
    float g_freq_history[5] = {};
    int   g_freq_idx        = 0;
    float g_ema_freq        = 0.f;
    float g_ema_conf        = 0.f;
    bool  g_smoothing_init  = false;
}

extern "C" {
    EXPORT YinResult audio_poll_pitch(int32_t sample_rate) {
        // ...
    }
}
```

## 问题四（潜在）：Isolate 里重建 `AudioBridge` 单例

`audio_engine.dart` 的 `_pollLoop` 是在 spawned Isolate 里执行的：

```dart
static void _pollLoop(SendPort sendPort) {
  final bridge = AudioBridge.instance;  // 新 Isolate，_instance == null，重新 open DLL
```

Dart Isolate 内存隔离，`_instance` 在新 Isolate 里是 null，会重新 `DynamicLibrary.open()`。Windows 上同一进程加载同一 DLL 会复用，所以 C 全局状态（`g_ring` 等）是共享的，**目前侥幸没问题**，但依赖这个行为不稳定。

更稳的做法是把 `DynamicLibrary` 的 handle 通过 `SendPort` 传入 Isolate，或者直接用 `Isolate.run` + 返回值而非长跑 Isolate。不过这个不是当前麦克风没数据的原因，可以二期处理。

## 快速验证步骤

改完之后加一个诊断日志确认数据链路通了：

```cpp
// audio_bridge.cpp — audio_poll_pitch 里临时加
EXPORT YinResult audio_poll_pitch(int32_t sample_rate) {
    float buf[4096];
    size_t n = g_ring.read(buf, 4096);
    // ← 加这一行，能看到 n > 0 就说明麦克风数据进来了
    OutputDebugStringA(("poll: n=" + std::to_string(n) + "\n").c_str());
    ...
}
```

用 DebugView 或 Visual Studio 输出窗口观察，`n` 持续为 0 说明 WASAPI 那边还有问题，`n > 0` 说明数据流通了。
