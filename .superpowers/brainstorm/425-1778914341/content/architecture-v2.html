架构迭代 — 两种改进方案

核心改进目标：统一音频管线 + 清晰的模块边界

当前方案的问题

痛点

  • 音频分两条路径：WebView(Tone.js) + FFI(PortAudio)，无统一时钟
  • WebView 音频延迟不可控，与原生音准检测无法对齐
  • "自动跟谱"等功能需要音频 ↔ 乐谱精确同步，分裂架构做不到
  • 模块边界模糊，代码容易耦合
A

统一音频 + 分层架构（推荐）

所有音频走一条 C 管线（PortAudio + FluidSynth + Aubio），由 AudioEngine 统一调度。WebView 只做乐谱渲染（纯视觉），不做音频。App 内部按 Clean Architecture 分层：UI → State → Domain → Data → Infrastructure。

B

插件化工具架构

每个工具（调音器、节拍器、乐谱视图、练习计划...）都是一个插件，实现统一的 ToolPlugin 接口。主框架只负责工具注册、布局和全局服务（音频引擎、数据层）。扩展性强，但前期设计成本高。

维度 A — 统一音频+分层 B — 插件化
音频同步 ✅ 统一AudioEngine，一个时钟源 ✅ 同样可统一
前期开发速度 ✅ 较快，分层自然 ⚠ 接口设计需要更多前期思考
后期扩展 ⚠ 新功能需要跨层修改 ✅ 加新工具只需写新Plugin
适合项目规模 ✅ 1-2人，中大型项目 ⚠ 更适合大型/团队项目
过度设计风险 ✅ 低，结构自然 ⚠ 有，前期抽象成本高

推荐 A — 统一音频 + Clean Architecture：解决了原方案的核心问题（音频同步），不过度设计，模块边界清晰。选好后告诉我。
